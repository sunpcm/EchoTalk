## 一、架构决策速查（全局，不因 Phase 改变）

| 决策         | 选择                               | 不选原因                                                                                  |
| ------------ | ---------------------------------- | ----------------------------------------------------------------------------------------- |
| 传输协议     | **LiveKit WebRTC**                 | WebSocket 基于 TCP，丢包导致队头阻塞，音频卡顿，且无原生媒体时间戳同步                    |
| 免费版对话   | **Deepgram Flux + LLM + Cartesia** | 目标 <800ms 端到端；Cartesia 状态空间模型首包 40-90ms，填补级联延迟                       |
| Premium 对话 | **Hume EVI 3 S2S**                 | 原生情绪感知，跳过文本中间层；Hume 专为缓解学习焦虑设计，比 OpenAI Realtime API 便宜 4-8x |
| 发音反馈     | **Needleman-Wunsch 音素对齐**      | 词级别颜色高亮太粗糙；音素对齐能精确展示 substitution/deletion/insertion 错误，反馈可解释 |
| 错误纠正策略 | **隐性重述（Recast）**             | 直接指出语法错误会加剧外语焦虑（FLA），Recast 在对话中自然带出正确形式                    |
| Pro 发音评估 | **ELSA Speech API**                | 非母语口音数据量最大，口音宽容度高，降低"发音正确被误判为错误"导致的用户流失              |
| 知识追踪     | **BKT（贝叶斯知识追踪）**          | 每个技能点独立维护掌握概率，可直接驱动 RAG 检索和 Prompt 定制                             |
| 数字人基础版 | **Sync.so**                        | API 优先，按量计费，适合 MVP；Premium 再升级 Tavus Phoenix-3（1080p 微表情）              |

## 三、Phase 开发前检查清单

### Phase 1 开始前

- [ ] `.env` 中填好 LiveKit / Deepgram / Cartesia / OpenRouter 四组 Key
- [ ] 本地 PostgreSQL 启动，`alembic upgrade head` 建表
- [ ] LiveKit 开发环境：使用 LiveKit Cloud 免费套餐或本地 `livekit-server`
- [ ] 理解 LiveKit Agent 模型：Agent 以服务端身份加入 Room，订阅用户音频轨，处理后推送合成音频

**核心链路**：`LiveKit Room → Agent 收音频 → Deepgram STT → OpenRouter LLM → Cartesia TTS → LiveKit 推音频`

---

### Phase 2 开始前

- [ ] Phase 1 验收：对话延迟 <800ms，`transcripts` 表有数据
- [ ] Redis 启动（Celery broker），`celery worker` 可正常消费任务
- [ ] 了解 Needleman-Wunsch 算法（打分矩阵：match=+2, mismatch=-1, gap=-1）
- [ ] 备好 CMU Pronouncing Dictionary（`pip install pronouncing`）

**BKT 公式记忆**：`p_posterior = P(mastery) * P(obs|mastery) / P(obs)`，更新后加 transit 概率

---

### Phase 3 开始前

- [ ] Phase 2 验收：BKT 数据写入正常，发音高亮前端可用
- [ ] Hume EVI 3 接入方式：**WebSocket**（非 REST），需在会话开始时建连并保持
- [ ] 情绪分析依赖：`pip install librosa soundfile scikit-learn`
- [ ] Sync.so 注意：API 有生成延迟（约 200-500ms），需在音频推送前预缓冲

**声画同步关键点**：Sync.so 返回视频帧时携带音频时间戳，前端用 `requestAnimationFrame` 对齐，不要用 `setTimeout`

---

### Phase 4 开始前

- [ ] Phase 3 验收：Premium 管线可用，情绪分析上线
- [ ] 语料库至少 500 条有标签文档（`skill_tags` + `difficulty_cefr` 字段必填）
- [ ] ChromaDB 本地启动：`docker run -p 8000:8000 chromadb/chroma`
- [ ] Embedding 模型选择：本地 `all-MiniLM-L6-v2`（免费，384 维）或 OpenAI `text-embedding-3-small`（$0.02/1M tokens）

**Krashen i+1 过滤规则**：用户当前 CEFR 水平 + 0.5 ~ +1.5，超出范围的材料不推送

## 四、关键技术实现速查

### 4.1 Needleman-Wunsch 音素对齐输出格式

```python
# 前端消费格式
[
  {"position": 0, "phoneme": "θ", "expected": "θ", "actual": "s", "type": "substitution"},
  {"position": 1, "phoneme": "ɪ", "expected": "ɪ", "actual": "ɪ", "type": "correct"},
  {"position": 2, "phoneme": "n", "expected": "n", "actual": None,  "type": "deletion"},
]
# type: "correct" | "substitution" | "deletion" | "insertion"
```

### 4.2 BKT 默认参数（冷启动）

```python
BKTParams(
    p_init     = 0.1,   # 初始掌握概率（保守估计）
    p_transit  = 0.2,   # 单次练习后新学会的概率
    p_slip     = 0.1,   # 已掌握但答错（失误率）
    p_guess    = 0.2,   # 未掌握但猜对
)
```

掌握判定阈值：`p_mastery > 0.95` 时视为该技能已掌握，从弱项列表中移除。

### 4.3 LLM System Prompt 结构（教练模式）

```
你是一位友善、有耐心的 AI 英语口语教练。

[场景] {scenario_description}

[用户当前薄弱技能] {weak_skills_from_bkt}  ← 来自 BKT 输出

[纠错策略] 使用隐性重述（Recast）：不直接打断用户，在你的回复中自然地用正确形式重复用户所说的内容。

[情绪指令] 当前用户焦虑指数：{anxiety_level}
- anxiety_level > 0.6：切换为鼓励模式，简化语言，增加正向反馈，避免复杂语法术语
- anxiety_level <= 0.6：正常教学模式，可适度提出挑战

[练习材料] {rag_retrieved_material}  ← Phase 4 后启用
```

### 4.4 情绪分析阈值

```python
# 基于音频特征的规则判断（librosa 提取后）
anxiety_level:
  > 0.6  → 鼓励模式（Recast，降低复杂度）
  0.3-0.6 → 正常教学
  < 0.3  → 可适当挑战（提高难度）

cognitive_load:
  "high" 触发条件：犹豫词（uh/um/er）频率 > 3次/分钟 或 平均停顿 > 2秒
```

## 六、异步处理边界（Celery 任务）

每次会话结束后触发的异步处理顺序：

```
1. analysis_tasks.analyze_session(session_id)
   ├── Whisper Large 精准转录（慢，不阻塞对话）
   ├── 发音对齐 → 写 pronunciation_assessments
   └── 语法错误检测 → 写 grammar_errors

2. knowledge_tasks.update_knowledge(session_id)
   └── 读取 pronunciation/grammar 错误 → BKT 更新 → 写 knowledge_states

3. report_tasks.generate_report(user_id)  ← 每周 Celery beat 触发
   └── 汇总 7 天数据 → OpenRouter Batch API → 写报告表
```

> 注意：主对话管线（Phase 1 的 free_pipeline/premium_pipeline）**不等待**上述任务，全部异步。
